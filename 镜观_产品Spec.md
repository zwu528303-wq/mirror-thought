# 镜观 产品规格说明书（MVP）

版本：v0.1  
日期：2026-05-30  
状态：MVP 开发规格草案  
产品暂名：镜观  

## 1. 产品概述

### 1.1 一句话定义

镜观是一个在线思想分析工具，通过苏格拉底式追问，帮助用户把模糊的困惑整理为核心信念、理由前提与冲突关系。

### 1.2 核心定位

镜观提供的是哲学意义上的「思想分析」，不是心理咨询、情绪支持、人生建议或知识问答。

产品的核心任务不是替用户做决定，而是帮助用户看清：

- 自己到底同时相信了什么；
- 这些信念分别依赖哪些理由或前提；
- 信念之间为什么产生张力、冲突或难以整合；
- 哪些概念、价值排序或事实判断需要进一步澄清。

### 1.3 核心价值主张

- 不提供答案，而是帮助用户看清自己思想的结构。
- 不提供情感安慰，而是提供逻辑层面的追问与澄清。
- 不评判用户信念的对错，而是协助用户检验信念之间的关系。
- 将「分析即治疗」的哲学咨询理念产品化，让用户随时获得一个严肃、克制、苏格拉底式的对话伙伴。

### 1.4 产品边界

镜观不是：

- 心理咨询产品；
- 心理危机干预工具；
- 情绪陪伴产品；
- 价值判断或人生建议工具；
- 哲学知识问答产品；
- 社区或论坛。

镜观处理的对象是「惑」：多个信念、判断、价值或前提之间难以整合的关系问题。

## 2. 目标用户与场景

### 2.1 目标用户

核心用户为 18-40 岁、对自身思想或观念感到困惑的普通人。典型用户包括大学生、青年职场人、哲学兴趣者，以及在重大选择前希望认真梳理想法的人。

用户通常具备以下特征：

- 脑子里有很多想法，但说不清它们之间的关系；
- 面临重要决定时感到价值冲突；
- 持有某个信念，但无法清楚说明为什么；
- 不满足于被安慰或被直接建议，希望获得逻辑层面的反馈。

### 2.2 不适合用户

以下用户不属于本产品服务对象：

- 正在经历心理健康危机的人；
- 有自伤、伤人、强烈绝望或严重情绪崩溃信号的人；
- 主要寻求情绪安慰、心理治疗或临床建议的人；
- 只想查询知识、资料或概念解释的人。

### 2.3 典型用户故事

| 编号 | 情境 | 用户想做什么 | 期望结果 |
| --- | --- | --- | --- |
| U1 | 我面临一个重大决定，内心矛盾 | 输入困惑，让 AI 帮我梳理背后的信念和价值排序 | 看清矛盾所在，做出更有底气的选择 |
| U2 | 我长期被某个人生问题困住 | 通过追问检验自己的假设和前提 | 发现困惑来自哪些未被检视的前提 |
| U3 | 我的某个信念被别人质疑 | 提交这个信念，让 AI 帮我检查理由结构 | 更清楚地理解和表达自己的立场 |

## 3. MVP 范围

### 3.1 MVP 必须交付

MVP 只交付以下四件事：

1. 首页定位说明：用户明确知道这是思想分析，不是心理咨询。
2. 多轮苏格拉底式对话：AI 先映射、再追问、再逐步澄清信念结构。
3. 文字版信念结构总结：MVP 不做可视化思想地图。
4. 危机中断机制：当用户出现心理危机信号时，停止思想分析并引导至专业支持资源。

### 3.2 MVP 明确不做

MVP 不包含：

- 可视化思想地图；
- 登录和注册；
- 历史记录；
- 结果保存或分享；
- PDF 导出；
- 付费系统；
- 社区、评论、用户互动；
- 长期用户档案或画像。

### 3.3 二期方向

二期可以在 MVP 验证后增加：

- 可视化思想地图；
- 信念节点与冲突关系图；
- 对话历史记录；
- 结果保存、复制和分享；
- 用户账号；
- PDF 或 Markdown 导出。

## 4. 核心概念

### 4.1 什么是「惑」

「惑」不是单一错误信念，而是多个信念之间的关系问题。

常见形式包括：

- 事实信念之间的不一致：我相信 A，也相信 B，但 A 和 B 似乎矛盾。
- 价值判断之间的冲突：我觉得应该 X，但也觉得应该 Y，无法取舍。
- 「是」与「应当」之间的张力：现实是 P，但我觉得应当是 Q。
- 信念与理由之间的断裂：我持有某个信念，但无法说明它的理由。

### 4.2 思想分析的基本原则

AI 不直接纠正用户的某个信念，而是帮助用户看清信念之间的结构。它的主要行为是：

- 映射：复述用户的困惑，确认理解是否准确。
- 追问：针对具体内容提出问题，澄清信念、理由、前提和概念。
- 识别：发现并指出信念之间的冲突或张力。
- 呈现：在适当时机将已经梳理出的信念结构总结出来。

## 5. 页面与信息架构

### 5.1 页面清单

| 页面 | MVP 是否包含 | 核心功能 |
| --- | --- | --- |
| 首页 | 是 | 产品定位、边界说明、示例对话、开始入口 |
| 使用须知弹窗 | 是 | 用户确认理解「这是思想分析，不是心理咨询」 |
| 对话页 | 是 | 输入困惑、多轮对话、AI 追问、阶段性总结 |
| 思想地图页 | 否 | 二期可视化信念结构 |
| 历史记录页 | 否 | 二期账号系统后实现 |
| 关于页 | 可选 | MVP 可先在首页底部简短说明 |

### 5.2 首页

首页目标是在 30 秒内让用户理解产品定位和边界。

必须包含：

- 产品名：镜观；
- 主标题：把你的困惑说清楚；
- 副标题：不提供答案，只帮你想得更深；
- 定位声明：这里提供的是思想分析，不是心理咨询；
- CTA：开始一次对话；
- 简短示例对话；
- 危机与资质声明。

首页不应做成营销型落地页。视觉上应保持安静、克制、专注，像一个适合思考的空间。

### 5.3 使用须知弹窗

用户第一次点击「开始一次对话」时，展示极简确认弹窗。

文案建议：

> 镜观提供的是思想分析，不是心理咨询或危机干预。它不会替你做决定，也不会提供医疗或心理治疗建议。

按钮：

- 我理解，开始对话
- 返回

用户确认后进入对话页。同一浏览器可记住确认状态，避免每次重复弹窗。

### 5.4 对话页

对话页是 MVP 的核心界面。

必须包含：

- 对话消息区；
- 底部固定输入框；
- 用户与 AI 的消息样式区分；
- AI 追问的视觉强调；
- 阶段性总结按钮或提示；
- 危机状态覆盖页或中断提示。

MVP 不要求实现右侧信念列表，但代码结构应允许后续从 AI 输出中提取信念并展示。

## 6. 核心用户流程

### 6.1 首次使用流程

1. 用户进入首页。
2. 用户阅读产品定位，理解这是思想分析，不是心理咨询。
3. 用户点击「开始一次对话」。
4. 系统展示使用须知弹窗。
5. 用户点击「我理解，开始对话」。
6. 用户进入对话页。
7. 用户输入困惑、问题或矛盾。
8. AI 先复述用户内容并请求确认。
9. AI 根据用户回答进行多轮追问。
10. 当达到可收束状态时，AI 主动建议生成文字版信念结构总结。
11. 用户可继续对话，或查看本轮总结。

### 6.2 回访流程

MVP 不做账号与历史记录。用户刷新页面后，会话可以丢失。后续二期再支持保存与恢复。

## 7. 对话完成态

### 7.1 为什么需要完成态

苏格拉底式对话可以无限延伸，但产品必须让用户知道「这次对话已经获得了一个阶段性结果」。因此 MVP 需要定义对话完成态。

### 7.2 可以收束状态

当 AI 已经识别出以下内容时，会话进入「可以收束」状态：

1. 至少 2 个核心信念；
2. 至少 1 组信念之间的张力、冲突或难以整合之处；
3. 用户对 AI 的初始理解有过至少一次确认；
4. 已经完成至少 5 轮有效对话。

进入该状态后，AI 可以提示用户：

> 目前我们已经整理出一个初步结构。要不要我先把这次对话中出现的核心信念和它们之间的张力总结出来？

### 7.3 默认收束规则

- 第 5 轮后：AI 可以开始判断是否具备总结条件。
- 第 8 轮后：AI 应主动提示是否生成阶段性总结。
- 第 12 轮后：AI 必须建议阶段性总结，避免无限追问。

用户可以选择继续对话，但系统应持续允许用户随时生成总结。

### 7.4 MVP 总结形式

MVP 的终点是「文字版信念结构总结」，不是可视化思想地图。

总结格式建议：

```markdown
我们目前整理出的结构是：

1. 你的核心信念
   - 信念 A：……
   - 信念 B：……
   - 信念 C：……

2. 它们之间的张力
   - 张力 1：A 与 B 在……方面存在冲突。

3. 仍需澄清的问题
   - 问题 1：……

4. 这次对话暂时抵达的位置
   - 你现在似乎不是缺少答案，而是需要进一步区分……
```

AI 在总结中不能给出结论性建议，不能告诉用户应该怎么选。

## 8. AI 行为规范

### 8.1 AI 角色

AI 是一位哲学思想分析师，扮演苏格拉底式对话者。

AI 不是：

- 心理咨询师；
- 情绪陪伴者；
- 答案提供者；
- 人生导师；
- 百科知识库。

### 8.2 每轮回应原则

AI 的每轮回应应尽量包含以下动作中的一个或多个：

- 映射：用自己的话复述用户的困惑；
- 追问：提出一个指向信念、理由、前提或概念的问题；
- 识别：指出信念之间的张力；
- 呈现：阶段性整理已经出现的信念结构。

第一轮必须先映射，再追问。不能直接给建议或直接分析结论。

### 8.3 第一轮规则

第一轮允许使用固定结构，但不固定措辞。

第一轮必须包含：

1. 感谢用户带来问题；
2. 用自己的话复述用户困惑；
3. 请求用户确认理解是否准确；
4. 最多提出一个澄清问题。

示例：

> 谢谢你把这个问题带来。让我先确认我理解得是否准确：你一方面认为……，另一方面又觉得……，这两者在当前选择中发生了冲突。我理解得对吗？如果是，我想先问：当你说「……」时，你具体指的是什么？

### 8.4 第二轮及之后

从第二轮开始，AI 不得使用固定模板追问。每次追问都必须针对用户刚刚提供的具体内容。

AI 应优先追问：

- 用户使用但尚未澄清的关键概念；
- 某个信念背后的理由；
- 两个信念之间的关系；
- 事实判断与价值判断的区别；
- 用户未说明但影响判断的前提。

### 8.5 AI 绝对不能做的事

AI 不得：

- 说「你应该……」；
- 给出人生选择建议；
- 替用户做价值排序；
- 判断用户某个信念是对或错；
- 以情绪安慰作为主要回应方式；
- 把哲学问题改写成心理诊断；
- 在危机场景中继续思想分析；
- 使用通用追问模板敷衍用户。

### 8.6 回应风格

- 语气：温和、平等、尊重、克制；
- 句式：多用问句，少用结论性陈述；
- 长度：每轮不超过 150 中文字；
- 追问数量：通常 1 个，最多 2 个；
- 语言：跟随用户使用的语言。

## 9. AI 输出协议

### 9.1 推荐结构化返回

为了支持危机中断、阶段总结和后续思想地图扩展，后端应要求 AI 返回结构化数据。

推荐 JSON 结构：

```json
{
  "response_type": "normal",
  "risk_level": "none",
  "message": "谢谢你把这个问题带来。让我先确认我理解得是否准确……",
  "detected_beliefs": [
    "陪伴家人很重要",
    "追求个人发展很重要"
  ],
  "detected_tensions": [
    "家庭陪伴与个人发展在地点选择上发生冲突"
  ],
  "can_summarize": false,
  "should_summarize": false
}
```

### 9.2 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| response_type | string | `normal`、`summary` 或 `crisis` |
| risk_level | string | `none`、`low`、`high` |
| message | string | 展示给用户的 AI 回复 |
| detected_beliefs | string[] | 当前已识别出的核心信念 |
| detected_tensions | string[] | 当前已识别出的冲突或张力 |
| can_summarize | boolean | 是否已经进入可以收束状态 |
| should_summarize | boolean | 是否应主动建议总结 |

### 9.3 前端展示规则

- 当 `response_type = normal`：展示普通 AI 消息。
- 当 `response_type = summary`：展示结构化文字总结。
- 当 `response_type = crisis`：停止普通对话，展示危机提示。
- 当 `can_summarize = true`：可展示「生成本轮总结」入口。
- 当 `should_summarize = true`：AI 主动提示用户是否总结。

## 10. 危机识别与中断机制

### 10.1 设计原则

危机识别不能只依赖前端关键词检测。危机信号经常是语义性的，不一定包含「自杀」「伤害」「崩溃」等明确词汇。

推荐机制：

1. AI system prompt 内置危机识别规则；
2. AI 每轮返回结构化字段 `response_type` 和 `risk_level`；
3. 前端根据结构化字段展示危机提示；
4. 前端关键词检测只作为第二道保险，不作为主判断逻辑。

### 10.2 危机信号

以下情况应触发危机流程：

- 用户提到自伤、自杀、伤害他人；
- 用户表达强烈绝望或无法保证自身安全；
- 用户出现严重情绪崩溃，明显无法继续理性对话；
- 用户表达带有告别、计划、手段或时间点的高风险内容；
- 在具体语境中，用户表现出放弃求助或危险平静的信号。

### 10.3 危机状态下 AI 行为

危机场景下，AI 必须：

- 停止思想分析；
- 不继续追问信念结构；
- 不进行哲学讨论；
- 不评价用户的想法；
- 引导用户联系专业心理健康支持或当地紧急服务。

示例文案：

> 我注意到你现在可能正在经历的不只是思想上的困惑。思想分析在这个时刻可能不是你最需要的。请优先联系你所在地的专业心理健康支持机构、可信任的人，或当地紧急服务。

具体危机资源应按部署地区配置，不写死在 prompt 中。

## 11. 功能需求

### F-001 首页定位说明

首页必须清楚说明产品是思想分析，不是心理咨询。

验收标准：

- 用户在首屏即可看到定位声明；
- 首页出现「不是心理咨询」或等价表达；
- CTA 文案清晰指向开始对话。

### F-002 使用须知确认

用户首次进入对话前必须确认理解产品边界。

验收标准：

- 首次点击 CTA 后出现确认弹窗；
- 用户点击确认后才能进入对话；
- 同一浏览器可记住已确认状态。

### F-003 对话输入

用户可以输入一段困惑、矛盾、问题或信念。

验收标准：

- 输入框支持长文本；
- 空内容不可提交；
- 提交后用户消息显示在对话区；
- AI 响应时有加载状态。

### F-004 第一轮映射

用户第一次提交后，AI 必须先复述并确认理解。

验收标准：

- AI 第一轮不能直接给建议；
- AI 第一轮必须包含对用户困惑的复述；
- AI 第一轮必须请求确认理解是否准确。

### F-005 多轮追问

AI 根据用户回答继续追问，逐步识别信念和张力。

验收标准：

- 每次追问针对用户具体内容；
- AI 每轮不超过 150 中文字；
- AI 通常只问 1 个问题；
- AI 不使用「你应该……」类建议句式。

### F-006 信念结构提取

后端应从 AI 响应中维护当前会话的信念结构状态。

验收标准：

- 每轮 AI 返回 `detected_beliefs`；
- 每轮 AI 返回 `detected_tensions`；
- 前端无需在 MVP 展示完整列表，但数据结构应保留。

### F-007 阶段性总结

当会话达到收束条件时，AI 主动建议生成文字总结。

验收标准：

- 第 8 轮后 AI 主动提示是否总结；
- 第 12 轮后 AI 必须建议总结；
- 总结内容包含核心信念、张力、仍需澄清的问题；
- 总结不提供结论性建议。

### F-008 危机中断

当 AI 判断为危机场景时，系统停止普通思想分析。

验收标准：

- AI 返回 `response_type = crisis` 时，前端展示危机提示；
- 输入区可被禁用或弱化；
- 不继续生成思想分析追问；
- 危机提示不包含诊断，不包含哲学分析。

### F-009 错误处理

AI 请求失败时，应显示可理解的错误状态。

验收标准：

- 网络或模型错误时展示重试按钮；
- 不丢失用户已输入内容；
- 不展示技术堆栈给用户。

## 12. 非功能需求

### 12.1 终端优先级

桌面端优先。思想分析需要较长输入和阅读，桌面端体验优先于移动端。

移动端需做基本适配，但不作为 MVP 体验优化重点。

### 12.2 性能

- 首页首屏应快速加载；
- AI 响应期间展示明确加载状态；
- 对话区域应支持至少 12 轮消息流畅滚动。

### 12.3 隐私

MVP 不做登录和历史记录，因此默认不提供跨设备保存。

开发时应避免在前端持久化敏感对话内容，除非用户明确保存或后续版本加入账号体系。

### 12.4 可配置性

以下内容应通过配置管理：

- AI 模型名称；
- API key；
- 危机资源文案；
- 是否开启调试日志；
- 单轮最大 token；
- 最大建议轮数。

模型名称不要在 PRD 中写死。开发交付时应根据 OpenAI / Anthropic 官方文档确认当前可用模型。

## 13. 视觉与交互风格

### 13.1 整体调性

视觉调性应体现：

- 克制；
- 安静；
- 专注；
- 有呼吸感；
- 像一个思考空间，而不是商业 App。

### 13.2 建议风格

- 浅色背景，奶油白或暖灰为主；
- 少量深色用于强调；
- 避免鲜艳色彩；
- 字体清晰，中文阅读舒适；
- 页面留白充足；
- 对话界面减少干扰元素。

### 13.3 避免风格

应避免：

- 强科技感；
- 心理咨询网站常见的暖橙治愈风；
- 论坛式布局；
- 营销型大段宣传；
- 过多装饰性卡片；
- 让用户误以为这是心理治疗工具的视觉语言。

## 14. 关键文案

### 14.1 产品名

推荐暂定名：镜观。

理由：

- 有「镜子映照自身」的意象；
- 不强行绑定某个哲学传统；
- 比「逻各斯」更亲近；
- 比「思辨」更有品牌感；
- 比「哲问」更含蓄、有记忆点。

### 14.2 首页文案

主标题：

> 把你的困惑说清楚

副标题：

> 不提供答案，只帮你想得更深

定位声明：

> 这里提供的是思想分析，不是心理咨询。我们通过苏格拉底式追问，帮你澄清信念、检验理由、理清矛盾。

CTA：

> 开始一次对话

输入框 placeholder：

> 把你最近在思考的一个问题、一个矛盾，或一个困扰你的信念写下来……

底部声明：

> 本平台提供思想分析服务，不具备医疗或心理咨询资质。如您正在经历心理健康危机，请联系专业机构或当地紧急服务。

## 15. 验收标准

MVP 完成时必须满足：

- 用户能在首页 30 秒内理解这是思想分析，不是心理咨询；
- 用户第一次进入对话前完成使用须知确认；
- 用户输入困惑后，AI 第一轮必须复述并确认理解；
- AI 不说「你应该……」或给出结论性建议；
- AI 的追问针对用户具体内容，不是通用模板；
- 多轮对话后，用户能看到核心信念和张力被逐步澄清；
- 第 8 轮后系统能主动建议生成文字版总结；
- 危机信号触发时，系统停止思想分析并展示专业支持提示；
- 桌面端主流浏览器可正常运行；
- MVP 不要求实现可视化思想地图、登录或历史记录。

## 16. 给开发者的实现备注

### 16.1 推荐技术实现口径

前端可以先实现为单页应用：

- 首页；
- 对话页；
- 使用须知弹窗；
- 危机提示状态；
- 阶段性总结状态。

后端至少需要：

- 接收完整会话历史；
- 调用 AI 模型；
- 要求 AI 返回结构化 JSON；
- 保存当前会话中的信念、张力和轮数状态；
- 根据 AI 输出控制前端展示状态。

### 16.2 Prompt 管理

system prompt 不应散落在前端代码中。建议放在后端配置或单独 prompt 文件中，便于迭代。

### 16.3 日志

MVP 开发调试阶段可以记录匿名化日志，但不应在生产环境中无提示保存用户完整对话。

## 17. 开放问题

以下问题可在开发前或 MVP 验证后继续确认：

- 危机资源是否按中国大陆、海外或用户所在地分别配置？
- 是否需要用户手动复制总结，还是 MVP 提供复制按钮？
- 「镜观」是否作为最终品牌名，还是仅作为工作代号？
- 是否需要在首页提供更多示例对话？
- 是否需要允许用户选择「继续追问」或「生成总结」两种模式？
